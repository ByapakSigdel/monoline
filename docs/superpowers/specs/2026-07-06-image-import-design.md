# monoline v0.2.0 — image import + repo media — design spec

**Date:** 2026-07-06
**Status:** Approved (design review with project owner)
**Baseline:** v0.1.0 (commit 916a53e)

## 1. Overview

Two deliverables:

1. **Image import**: get a normal image (PNG/JPG/WebP/BMP/GIF) into monoline
   and convert it to color-accurate braille art on the canvas — as a
   background **bitmap layer** the user can draw and erase over. Four entry
   points: drop-onto-terminal, `i` dialog, `monoline photo.png`, and `v`
   (clipboard).
2. **Repo media**: replace the README's "screenshot coming soon" placeholder
   with real, reproducible media — two SVG screenshots of the live app and
   two animated GIFs — all generated by scripts committed to the repo.

Ships as **v0.2.0** (minor bump: new feature, backward-compatible files).

### Goals

- A dropped photo looks genuinely good: Floyd-Steinberg dithering, per-cell
  24-bit color, aspect-preserving fit.
- The image is a first-class document layer: undoable import, saved in
  `.mono.json`, present in ANSI and SVG exports, erasable *visually* without
  destroying the source data.
- Plain drawings keep full compatibility with monoline 0.1.0 files.
- All README media reproducible by re-running repo scripts.

### Non-goals (this release)

- Dithering-mode config (Bayer/threshold variants), brightness/contrast
  controls.
- Bitmap editing tools (move, scale, crop after import) — replace or undo
  only.
- Clipboard *polling* (import is an explicit `v` action).
- PNG raster export (still deferred).

### Color-accuracy limit (stated up front)

A braille cell's 8 dots share one terminal foreground color. "Color
accurate" therefore means **accurate per cell**: dot pattern carries
luminance detail (via dithering), cell color carries chroma. This is the
physical ceiling of the medium and the README says so.

## 2. Bitmap layer (data model)

New pure module `monoline/bitmap.py`:

```python
@dataclass
class Bitmap:
    width: int    # dots (matches the document's dot dimensions)
    height: int   # dots
    cells: Dict[Tuple[int, int], Tuple[int, str]]
    # (col, row) -> (bits 1-255 braille pattern, "#rrggbb" cell color)
```

Only non-empty cells appear in `cells` (an all-off cell is absent, never
`bits=0`, matching the §6 file validation); `bits` uses the same braille bit
layout as `raster.DOT_BITS`. Treated as immutable by convention — mutations
replace the whole object.

`monoline/document.py` changes:

- `Document.bitmap: Optional[Bitmap]`, default `None`.
- `set_bitmap(bitmap: Optional[Bitmap]) -> None` — pushes a `_SetBitmap
  (previous, new)` op; undo restores `previous`, redo re-applies `new`;
  clears the redo stack; sets `dirty`. Import, replace, and remove all go
  through this one method.
- `clear()` extends `_Clear` to `(previous_strokes, previous_bitmap)`:
  clearing removes strokes AND bitmap as one undo unit. `clear()` is a
  no-op only when there are no strokes AND no bitmap.

## 3. Rendering (compositing)

`raster.render_cells(strokes, width, height, bitmap=None)`:

1. **Seed** the dot grid from the bitmap: each set bit in each cell becomes
   a dot at `(cx*2 + dx, cy*4 + dy)` with sequence 0 and the cell's color
   (skip cells outside `width × height`).
2. Apply strokes in order exactly as today (pen seq 1..n sets dots, erase
   clears dots within radius).

Falls out for free: pen strokes render **over** the image
(higher sequence wins the cell color), and the **eraser visually erases
image dots** — non-destructively, since the bitmap object is untouched and
the erase stroke is an ordinary undoable stroke.

Call-site updates: `canvas.rebuild()` and `io.export_ansi` pass
`document.bitmap`.

**SVG export**: lit bitmap dots render beneath the stroke polylines as
`<circle cx=".." cy=".." r="0.55" fill="{cellcolor}"/>` (dot centers at
`x+0.5, y+0.5` in dot units). Erase strokes (background-colored, round-cap
polylines) already paint over both layers, keeping WYSIWYG. Size note: a
full-terminal photo lights ~10-20k dots → SVG around 1 MB; acceptable.

## 4. Converter

New module `monoline/imageconv.py` — depends on Pillow (new **hard runtime
dependency `pillow>=10`**; prebuilt wheels exist for the full OS × Python
support matrix).

```python
convert(img: PIL.Image.Image, dot_w: int, dot_h: int, background: str) -> Bitmap
```

Pipeline:

1. `ImageOps.exif_transpose` (respect camera orientation).
2. Convert to RGBA and alpha-composite over `background` (the document's
   background color), then to RGB.
3. Resize to fit inside `dot_w × dot_h` preserving aspect ratio
   (`scale = min(dot_w/w, dot_h/h)`, LANCZOS), centered — the offset places
   the image in the middle of the canvas; uncovered dots stay unlit.
4. Luminance (Pillow `convert("L")`) as floats; **Floyd-Steinberg
   error-diffusion** (plain left-to-right scan, mid threshold 127.5,
   standard 7/16-3/16-5/16-1/16 kernel, pure Python — dot grids are ≤ ~100k
   pixels, no NumPy needed). Dots dithered to white are **lit**
   (light-on-dark canvas).
5. Cell color: average RGB of that cell's *lit* pixels from the resized
   color image, formatted `#rrggbb`. Cells with no lit dots are absent.

Deterministic: same image + same canvas size → identical Bitmap.

## 5. Import UX (four doors)

`IMAGE_SUFFIXES = {".png", ".jpg", ".jpeg", ".webp", ".bmp", ".gif"}`

All doors funnel into one app method
`_import_image(source: path-or-PIL-Image)`: open/convert with the
document's dot dimensions and background → `document.set_bitmap(...)` →
`canvas.rebuild()` → `notify("imported {name}")`. Any
`UnidentifiedImageError`/`OSError`/`ValueError` → `notify(...,
severity="error")`; the app never crashes on a bad file.

- **Drop-onto-terminal**: dropping a file pastes its (possibly quoted)
  path. App-level `on_paste`: strip whitespace and surrounding quotes; if
  the text is an existing file with an image suffix → import. Otherwise
  ignore (dialogs keep native paste behavior — the app-level handler only
  sees pastes no focused widget consumed; verified empirically during
  implementation).
- **`i` key**: `TextPrompt("Import image:")` → path → import. Listed in
  the `?` help overlay.
- **CLI**: `monoline photo.png` — an image-suffixed argument starts an
  untitled document with `pending_import` set; the import runs right after
  the canvas receives its size (same first-layout mechanism as document
  sizing). Any non-image argument behaves exactly as in v0.1.0.
- **`v` key**: `PIL.ImageGrab.grabclipboard()` (imported lazily inside the
  action). Result: `Image` → import; list of file paths → import the first
  with an image suffix; `None`/unsupported → notify "no image in the
  clipboard". Works natively on Windows/macOS; on Linux Pillow requires
  `xclip`/`wl-paste` — failure notifies with that hint. No polling.

GIF sources import their first frame.

## 6. File format v2

`.mono.json` gains an optional `"bitmap"` field:

```json
"bitmap": {
  "width": 320, "height": 192,
  "cells": [[10, 4, 255, "#e08040"], [11, 4, 129, "#d07030"]]
}
```

- **`version` stays `1` when `bitmap` is absent** — plain drawings remain
  openable by monoline 0.1.0. Bitmap present → `"version": 2`.
- Loader accepts versions 1 and 2 (v1 must not contain `bitmap`).
- Validation (same hardening standard as v0.1.0, all failures →
  `MonolineError`): bitmap width/height ints `0 < v <= 100_000`; each cell
  entry is `[cx, cy, bits, color]` with `cx`/`cy` ints within the cell
  grid (`width//2 × height//4`), `bits` int `1..255`, `color` passing the
  existing `_color` hex check. Cell-count cap: at most `(width//2) *
  (height//4)` entries.
- `save()` writes the bitmap when present; round-trip is exact.

## 9. Testing

- `bitmap`/`document`: `set_bitmap` undo/redo (including replace and
  remove), `clear()` takes strokes+bitmap in one op, no-op rules.
- `imageconv` (synthetic PIL fixtures): solid white → all dots lit, cell
  color white; solid black → empty bitmap; left-white/right-black split →
  lit dots only left, correct centering; red/blue halves → per-cell colors
  match; horizontal gradient → lit-dot density increases monotonically
  across bands (dithering sanity); alpha image composites over background;
  EXIF-rotated fixture lands upright; aspect fit centers a tall image.
- `raster`: bitmap seeds beneath strokes; pen over bitmap wins cell color;
  erase stroke clears bitmap dots from the render while `document.bitmap`
  is unchanged.
- `io`: v2 round-trip exact; v1 files (no bitmap) still load; version-1
  written when no bitmap; validation rejections (bad bits, out-of-bounds
  cells, bad color, cell-count overflow) → `MonolineError`.
- App (Pilot): paste-of-image-path imports (tmp PNG fixture); `i` dialog
  imports; `v` with `ImageGrab.grabclipboard` monkeypatched to a PIL image
  imports, and to `None` notifies; CLI image argument imports after first
  layout; import is one undo step.
- Scripts: smoke test that both scripts run headlessly and produce
  non-empty files (marked slow; excluded from the default CI job only if
  runtime becomes a problem).

## 11. Risks & mitigations

- **Paste event routing differs across terminals/Textual versions** →
  implementation verifies App-level `on_paste` behavior empirically with
  Pilot; the `i` dialog is the universal fallback.
- **Clipboard support on Linux needs external tools** → graceful notify
  with the `xclip`/`wl-paste` hint; documented in README.
- **Large SVG exports with bitmaps** → acceptable at terminal scale (~1 MB);
  noted in README.
- **GIF size** → quantized palette, dot-grid frames compress well; scripts
  assert output size < 4 MB.
- **Old monoline opening v2 files** → version gate produces the clean
  "unsupported version" error added in v0.1.0 hardening; plain drawings
  still save as v1.
